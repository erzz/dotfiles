## Planning Standard

Use this structure for any non-trivial implementation plan:

1. **Summary** -- one or two sentences describing the intended outcome
2. **Files** -- the files or systems most likely to change
3. **Steps** -- short, concrete implementation steps
4. **Risks** -- hidden dependencies, ambiguity, or likely failure points
5. **Verification** -- specific checks, tests, or commands to run after implementation

## Output Discipline

- Keep plans concise and execution-ready
- Prefer file paths and bullets over long prose
- Do not include large code excerpts unless they are essential
- Reviewers should critique the same structure rather than invent a new format
